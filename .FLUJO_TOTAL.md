# Flujo total de la app (Hotelus PMS)

Este documento resume como funciona toda la aplicacion de punta a punta para retomarla despues y mejorarla sin perder contexto.

## 1) Stack y estructura base

- Frontend: React + Vite + React Router.
- UI: Tailwind + componentes propios (`Modal`, `Toast`, `StatusBadge`, etc.).
- Datos: dos modos de persistencia:
  - **Supabase** (si existen `VITE_SUPABASE_URL` y `VITE_SUPABASE_ANON_KEY`).
  - **Memoria/localStorage** con seeds si no hay Supabase.
- Multi-hotel: se usa `hotel_id` en servicios cuando Supabase esta activo.

Archivos clave:
- Entrada: `src/main.jsx`
- Enrutado: `src/App.jsx`
- Auth global: `src/contexts/AuthContext.jsx`
- Layout protegido: `src/components/ProtectedRoute.jsx` + `src/layouts/AppLayout.jsx`
- Servicios: `src/services/*.service.js`
- Reglas de estados: `src/constants/statuses.js`

## 2) Flujo de arranque (startup)

1. `main.jsx` monta React dentro de `BrowserRouter`.
2. `App.jsx`:
   - Muestra `SplashScreen` ~2200ms.
   - Envuelve todo con `AuthProvider`.
   - Carga paginas con `lazy` + `Suspense`.
3. Se definen rutas:
   - Publica: `/login`
   - Protegidas: `/dashboard`, `/rooms`, `/reservations`, `/guests`, `/calendar`
4. `ProtectedRoute` decide:
   - Si auth esta cargando -> loader.
   - Si Supabase activo y no autenticado -> redirige a `/login`.
   - Si autenticado pero perfil/hotel no cargado -> loader.
   - Si todo OK -> renderiza `AppLayout`.

## 3) Flujo de autenticacion y contexto

Archivo: `src/contexts/AuthContext.jsx`

### Que maneja
- `session`
- `loading` de sesion
- `profileLoaded`
- `hotelId`
- `signIn` / `signOut`
- `isAuthenticated`

### Secuencia real
1. `loadSession()` pide la sesion a Supabase.
2. Se escucha `onAuthStateChange` para mantener `session` sincronizada.
3. Si hay sesion, `loadProfile()` consulta `profiles.hotel_id`.
4. Ese `hotel_id` se guarda en:
   - estado del contexto (`hotelId`)
   - variable global `currentHotelId` (`src/lib/currentHotel.js`)
5. Los servicios leen `getCurrentHotelId()` para filtrar datos por hotel.

Si Supabase no esta configurado:
- no se bloquea el acceso,
- `isAuthenticated` se considera verdadero,
- se trabaja con datos locales en memoria/localStorage.

## 4) Flujo de navegacion y layout

Archivo: `src/layouts/AppLayout.jsx`

- Sidebar con secciones: Dashboard, Habitaciones, Reservas, Huespedes, Calendario.
- Header (`src/components/Header.jsx`) muestra:
  - titulo segun ruta,
  - fecha actual,
  - boton de cerrar sesion (solo si Supabase activo).
- El contenido de cada pagina se inyecta con `<Outlet />`.

## 5) Flujo de datos: servicios y fallback local

Servicios:
- `src/services/rooms.service.js`
- `src/services/reservations.service.js`
- `src/services/guests.service.js`

Patron comun:
1. Si `isSupabaseEnabled`:
   - CRUD sobre tablas (`rooms`, `reservations`, `guests`)
   - filtro por `hotel_id`
2. Si no:
   - se usa memoria + `localStorage` (`localStore.js`)
   - se inicializa con seeds (`src/data/*`)

Resultado: la app funciona en modo demo/offline y en modo productivo con Supabase.

## 6) Reglas de negocio centralizadas

Archivo: `src/constants/statuses.js`

Estados de reserva:
- `Confirmed`
- `CheckedIn`
- `CheckedOut`
- `Cancelled`

Estados de habitacion:
- `Disponible`
- `Ocupada`
- `Limpieza`

Funciones clave:
- `isActiveReservationStatus()`
- `getActiveReservationsForRoom()`
- `getActiveReservationsForGuest()`

Estas reglas se usan para validar eliminaciones y calcular ocupacion/disponibilidad.

## 7) Flujo por modulo

## Dashboard (`src/pages/Dashboard.jsx`)
- Carga reservas + habitaciones.
- Calcula metricas:
  - ocupacion, disponibles, limpieza
  - llegadas/salidas del dia
  - ingresos de hoy/mes
- Calcula ingresos con dos casos:
  - check-out del dia/mes (si no pago en check-in)
  - check-in pagado (se cuenta en ese momento, no se duplica en check-out)
- Muestra graficos de 7 dias (ocupacion e ingresos).

## Habitaciones (`src/pages/Rooms.jsx`)
- Carga habitaciones + reservas + huespedes.
- Permite crear/editar/eliminar habitaciones.
- Bloquea eliminacion si la habitacion tiene reservas activas.
- Abre `RoomQuickView` para acciones rapidas.

`RoomQuickView` (`src/components/RoomQuickView.jsx`):
- Detecta reserva activa/actual de la habitacion.
- Permite:
  - cambiar estado de habitacion,
  - check-in/check-out/cancelar de la reserva actual,
  - marcar si pago al hacer check-in.
- Al cambiar estado de reserva, sincroniza estado de habitacion:
  - `CheckedIn` -> `Ocupada`
  - `CheckedOut` -> `Limpieza`
  - `Cancelled` -> `Disponible`

## Reservas (`src/pages/Reservations.jsx`)
- Lista y filtra por:
  - busqueda (huesped/habitacion/id),
  - estado,
  - rango de fechas,
  - filtros rapidos (llegadas/salidas de hoy).
- Crear reserva valida:
  - campos obligatorios,
  - check-out >= check-in,
  - sin solapamiento de fechas para la misma habitacion.
- Acciones por reserva:
  - Check-in (pregunta si esta pago),
  - Check-out,
  - Cancelar,
  - Duplicar reserva,
  - Ver detalle.
- Detalle de reserva:
  - agregar/quitar cargos extra,
  - totalizar cargos,
  - mantener `paidAtCheckIn`.
- Exporta CSV e imprime.

## Huespedes (`src/pages/Guests.jsx`)
- CRUD de huespedes con validaciones:
  - nombre/email obligatorios,
  - email unico.
- Bloquea eliminacion si hay reservas activas asociadas.

## Calendario (`src/pages/Calendar.jsx`)
- Vista semanal o mensual de ocupacion por habitacion.
- Cada celda evalua si hay reserva en ese dia (`checkIn <= dia < checkOut`).
- Click en habitacion abre `RoomQuickView`.
- Desde quick view puede navegar al detalle de reserva en modulo Reservas.

## 8) Flujo principal de operacion diaria (end-to-end)

1. Usuario entra (login si Supabase activo).
2. Se carga `hotel_id` del perfil.
3. Navega por dashboard para ver estado actual.
4. Crea/actualiza reservas.
5. Check-in:
   - reserva pasa a `CheckedIn`,
   - habitacion pasa a `Ocupada`,
   - se registra si pago o queda pendiente.
6. Durante estadia:
   - se agregan cargos a la reserva.
7. Check-out:
   - reserva pasa a `CheckedOut`,
   - habitacion pasa a `Limpieza`.
8. Limpieza finalizada:
   - habitacion vuelve a `Disponible`.
9. Dashboard refleja automaticamente ocupacion e ingresos segun estados.

## 9) Puntos de mejora recomendados (para despues)

1. **Corregir detalle menor en `RoomQuickView`:**
   - hay un `9;` suelto dentro de `getCurrentReservation` que parece residual.
2. **Centralizar transiciones de estado**:
   - hoy se repiten reglas `CheckedIn/CheckedOut/Cancelled` en mas de un lugar.
   - conviene crear un helper unico para reducir inconsistencias.
3. **Manejar errores async de forma uniforme**:
   - algunas acciones actualizan UI optimista sin rollback en error.
4. **Agregar tests de reglas criticas**:
   - solapamiento de reservas,
   - calculo de ingresos,
   - cambios de estado reserva <-> habitacion.
5. **Documentar contrato de tablas Supabase**:
   - columnas esperadas, tipos y RLS minimo.

## 10) Resumen corto para retomar rapido

La app tiene un flujo claro: autenticacion -> carga de `hotel_id` -> modulos protegidos -> CRUD de habitaciones/huespedes/reservas -> transiciones operativas (check-in/check-out/limpieza) -> dashboard y calendario consumen esos estados para reportes y vista operativa.

Si queres mejorarla, lo mas importante es consolidar reglas de negocio en un solo lugar y cubrirlas con tests.
